# GenAI Tooling — Prompts, Output, and Critical Evaluation

This document answers the exercise's "Generative AI tools" section: the prompt(s) I would use to
generate the API, a representative sample of the output, and how I validated, corrected, and
hardened the AI's suggestions.

## Tool & overall approach

- **Tool:** Claude Code (Claude Opus 4.8), driving the whole repo.
- **Strategy — context before code.** I don't ask an AI to "build the whole API" from a cold start;
  that's how you get plausible-but-wrong architecture and banned libraries sneaking in. Instead I
  first establish a **context layer** the model must read on every interaction:
  - [`AGENTS.md`](../AGENTS.md) — an always-on index and single source of truth.
  - [`ai-context/`](../ai-context/) — `ARCHITECTURE.md`, `CONVENTIONS.md`, `RUNBOOK.md`, and
    `rules/` (task approach, definition of done, canonical files, security).
  - Bridge files (`CLAUDE.md`, etc.) just redirect tools to `AGENTS.md`.

  With the constraints (Clean Architecture, **no EF/Dapper/Mediator**, TDD, cookie-vs-token
  decisions, OpenAPI/Scalar) encoded once, every later prompt is grounded and the model generates
  *within* the architecture instead of inventing one. This is the core of how I get reliable output.
- **Orchestration — parallel agents with guardrails.** The implementation is built by multiple
  sub-agents running in **isolated git worktrees** (a thin foundation first, then feature slices in
  parallel), governed by two rules that keep AI output trustworthy: **nothing merges without my
  review of the diff**, and an agent that hits an ambiguous decision **stops and asks rather than
  guessing**.

## 1. The prompt I would use

This is the engineered scaffold prompt for the backend. Note the structure: **role → context
pointer → explicit constraints → the slice to build → guardrails → definition of done.** It is
deliberately specific about what *not* to do.

> **Role:** You are a senior .NET engineer working in this repository. Before doing anything, read
> `AGENTS.md` and the `ai-context/` layer; follow them.
>
> **Task:** Generate a RESTful task-management API as the first vertical slice: **create and list
> tasks** for the authenticated user. A task has `title`, `description`, `status`
> (`Todo`/`InProgress`/`Done`), and `dueDate`, and belongs to a user.
>
> **Architecture (must follow):** Clean Architecture across `Bla.Domain`, `Bla.Application`,
> `Bla.Infrastructure`, `Bla.Api`. Dependencies point inward. The Domain references nothing.
> Repositories are **ports** in `Application`, implemented in `Infrastructure`.
>
> **Hard constraints:**
> - **Do not use Entity Framework, Dapper, or Mediator.** Data access is **raw ADO.NET via Npgsql**
>   with hand-written, **parameterized** SQL, mapping `DbDataReader` to domain objects by hand.
> - Target .NET 10 but configure OpenAPI output as **3.0** (Angular generator compatibility).
> - **TDD:** write the failing xUnit test first, then the minimal code to pass.
> - Controllers contain no business logic; never expose domain entities over HTTP — use DTO records.
> - Auth: endpoints are authorized; the user id comes from the validated JWT, never from the request
>   body. Scope every query by `user_id`.
>
> **Deliver:** the `TaskItem` entity, `ITaskRepository` port, a `NpgsqlTaskRepository`, a
> `CreateTask`/`ListTasks` application service with validation, the `TasksController`, the SQL
> migration for the `tasks` table, and unit tests for the service and repository. Keep the build
> warning-clean.
>
> Stop after this slice; do not scaffold unrelated features.

**Why it's shaped this way:** the constraints that an AI most wants to violate (reach for EF, put
logic in controllers, return entities, build everything at once) are stated as explicit negatives,
and the work is scoped to **one vertical slice** so the output stays reviewable.

**Port (Application) — no I/O knowledge:**

```csharp
public interface ITaskRepository
{

    Task<IReadOnlyList<TaskItem>> ListByUserAsync(Guid userId, CancellationToken ct);
```

**Repository (Infrastructure) — raw ADO.NET via Npgsql, parameterized, manual mapping:**

```csharp
public sealed class NpgsqlTaskRepository : ITaskRepository
{
    private readonly NpgsqlDataSource _dataSource;
    public NpgsqlTaskRepository(NpgsqlDataSource dataSource) => _dataSource = dataSource;

    public async Task<Guid> AddAsync(TaskItem task, CancellationToken ct)
    {
        const string sql = """
            INSERT INTO tasks (id, title, description, status, due_date, user_id, created_at, updated_at)
            VALUES (@id, @title, @description, @status, @dueDate, @userId, @createdAt, @updatedAt)
            RETURNING id;
            """;
        await using var cmd = _dataSource.CreateCommand(sql);
        cmd.Parameters.AddWithValue("id", task.Id);
        cmd.Parameters.AddWithValue("title", task.Title);
        cmd.Parameters.AddWithValue("description", (object?)task.Description ?? DBNull.Value);
        cmd.Parameters.AddWithValue("status", task.Status.ToString());
        cmd.Parameters.AddWithValue("dueDate", task.DueDate);          // stored as UTC (timestamptz)
        cmd.Parameters.AddWithValue("userId", task.UserId);
        cmd.Parameters.AddWithValue("createdAt", task.CreatedAt);
        cmd.Parameters.AddWithValue("updatedAt", task.UpdatedAt);
        return (Guid)(await cmd.ExecuteScalarAsync(ct))!;
    }

    public async Task<IReadOnlyList<TaskItem>> ListByUserAsync(Guid userId, CancellationToken ct)
    {
        const string sql = """
            SELECT id, title, description, status, due_date, user_id, created_at, updated_at
            FROM tasks WHERE user_id = @userId ORDER BY due_date;
            """;
        await using var cmd = _dataSource.CreateCommand(sql);
        cmd.Parameters.AddWithValue("userId", userId);
        await using var reader = await cmd.ExecuteReaderAsync(ct);

        var tasks = new List<TaskItem>();
        while (await reader.ReadAsync(ct))
            tasks.Add(Map(reader));
        return tasks;
    }

    private static TaskItem Map(NpgsqlDataReader r) => new(
        id:          r.GetGuid(0),
        title:       r.GetString(1),
        description: r.IsDBNull(2) ? null : r.GetString(2),
        status:      Enum.Parse<TaskStatus>(r.GetString(3)),
        dueDate:     r.GetDateTime(4),
        userId:      r.GetGuid(5),
        createdAt:   r.GetDateTime(6),
        updatedAt:   r.GetDateTime(7));
}
```

**Controller (Api) — thin; user id from the token, not the body:**

```csharp
[ApiController]
[Route("api/tasks")]
[Authorize]
public sealed class TasksController(ITaskService tasks) : ControllerBase
{
    [HttpPost]
    public async Task<ActionResult<TaskResponse>> Create(CreateTaskRequest request, CancellationToken ct)
    {
        var created = await tasks.CreateAsync(User.GetUserId(), request, ct);
        return CreatedAtAction(nameof(GetById), new { id = created.Id }, created);
    }

    [HttpGet]
    public async Task<ActionResult<IReadOnlyList<TaskResponse>>> List(CancellationToken ct)
        => Ok(await tasks.ListAsync(User.GetUserId(), ct));
}
```

**Test (TDD, written first):**

```csharp
[Fact]
public async Task CreateAsync_PersistsTask_ForTheAuthenticatedUser()
{
    var repo = Substitute.For<ITaskRepository>();
    var sut = new TaskService(repo);
    var userId = Guid.NewGuid();

    await sut.CreateAsync(userId, new CreateTaskRequest("Write README", null, TaskStatus.Todo, _due));

    await repo.Received(1).AddAsync(
        Arg.Is<TaskItem>(t => t.UserId == userId && t.Title == "Write README"), Arg.Any<CancellationToken>());
}
```

## 3. How I validated, corrected, and handled edge cases

### Validated the AI's suggestions

- **Checked the constraint against the source, not the AI's paraphrase.** The AI's first instinct
  was an ORM. I went back to the PDF: it bans three *named libraries* (EF, Dapper, Mediator) — it
  never says "no ORM." That distinction changed the option space (a document-DB driver would be
  allowed too) and made the decision deliberate rather than assumed.
- **Pressure-tested the data-store choice.** When the AI proposed MongoDB for "modern storage," I
  asked it to argue *against* itself. The honest answer — the data is mildly relational
  (tasks belong to users) — led me to **PostgreSQL + raw ADO.NET (Npgsql)**: satisfies the ban,
  maximizes the "no comfort layer" signal, and keeps Clean Architecture intact (swapping the store
  touches one repository class).
- **Verified framework specifics instead of trusting memory.** The AI initially claimed ".NET 9+
  emits OpenAPI 3.1." I checked: .NET **9** defaults to 3.0; the 3.1 default arrived in .NET **10**
  — the exact version I'm on. Verifying the version detail turned a vague worry into a concrete fix.

### Corrected or improved the output

- **OpenAPI 3.1 vs. the client generator.** .NET 10's native OpenAPI emits 3.1 (JSON Schema
  2020-12), which `openapi-generator`'s `typescript-angular` handles poorly. Fix: pin the document
  to **3.0** (`options.OpenApiVersion = OpenApiSpecVersion.OpenApi3_0`) — keep .NET 10, avoid a
  demo-day failure in the frontend pipeline.
- **Rejected over-engineered auth.** The AI was happy to build `httpOnly` cookies + rotating,
  revocable refresh tokens + CSRF. For a take-home that's failure surface, not value. I switched to
  a **pragmatic** single ~1h bearer token in `localStorage`, and **documented the upgrade path** so
  the security knowledge is on record without the plumbing. (Right-sizing AI ambition is itself a
  validation skill.)
- **Cut redundancy and drift.** AI-generated docs repeated the same facts across files. I
  consolidated each fact to a single canonical home with pointers, so the docs can't drift out of
  sync as the code evolves.

### Handled edge cases, authentication, and validation

- **Authentication:** JWT bearer; the user id is read from the validated token, **never** from the
  request body. Every task query is scoped by `user_id` so one user can't touch another's data.
- **Authorization leaks:** distinguish `404` (not found) from `403` (forbidden) so the API doesn't
  reveal that another user's resource exists; `401` for missing/invalid token.
- **Validation:** inbound DTOs validated (required fields, lengths, `Status` is a known enum,
  `dueDate` sanity); failures return `400` with `ProblemDetails`.
- **Injection:** all SQL is **parameterized** — no string concatenation — closing SQL injection.
- **CORS:** the SPA (`:4200`) and API (`:8080`) are different origins; CORS allows the specific SPA
  origin (not `AllowAnyOrigin`), which also prevents the console errors the rubric penalizes.
- **Data correctness:** timestamps stored as **UTC** (`timestamptz`); nullable handled explicitly
  rather than silenced with `!`.

Full detail lives in [`ai-context/rules/security.md`](../ai-context/rules/security.md) and
[`ai-context/ARCHITECTURE.md`](../ai-context/ARCHITECTURE.md).
